JavaScript中的作用域和作用域链

一、作用域种类

目前，标准给了我们三种作用域，分别是全局作用域、函数作用域和块作用域。其实前两种作用域也可以看作是一种“块作用域”，那么为什么还有其存在的必要呢？历史原因。但其有着一些特殊的表现，使得程序员在编写JavaScript的时候可能出现一些莫名其妙的问题。本质上，就是没有搞清楚其设计的特殊之处，而作用域就是一大特殊之处，其坑非浅。

1、全局作用域与函数作用域

全局作用域与函数作用域是ES5标准及以前的产物，当不存在块作用域时，他们的一些行为被认为是非常合理的。但是当块作用域产生后，他们的一些情况方法和实际表现就有点难以理喻了。特别是函数作用域，其往往和闭包（closure）、上下文等概念一起出现，使得一些网络上的一些为了解释某种性质的示例代码同时包含了多种JavaScript语言特性，而经常搞混。后面会谈及函数作用域与闭包的问题，而上下文及其相关联的this指向问题由另一篇文章专门研究。

首先谈及的是全局作用域。语言规范定义的全局作用域是不在函数作用域内的最外层作用域。其实在我们实际编程中的情况比较复杂，repl环境、浏览器环境倒还好说，但是解释编译脚本文件的情况让人摸不着头脑——到底是全局作用域还是函数作用域？毕竟我们都知道，在Node.js现有的模块机制下，文件外层是被包裹了函数的外衣的。既然是在函数之下那自然是函数作用域了。但是，感觉上那是自己所编写代码的最外层，理应有“全局”的名分，然而不同文件的“全局”竟然各不相同——在不同文件的“全局作用域”下声明的变量无法直接被调用。因此，我更愿意把这样的作用域称为“模块作用域”，而称文件下不使用关键字声明的真正的全局变量是在全局作用域中的。幸好无论是何种叫法，都不影响实际的运行效果。

把这种作用域的称呼问题放在一边，探究一下其有哪些违反常理的地方。

三、作用域链

JavaScript在运行时中，会维护这么几种环境，其中就有作用域链和调用堆栈。

1、概念区别

上下文是运行时产生的，由函数的调用决定。不同上下文根据调用关系形成调用堆栈。因此可以说调用堆栈记录了函数的调用位置、局部变量等信息。

作用域在不存在动态作用域（with语句）的情况下是静态的，即词法作用域是由定义时决定的。不同作用域根据父子逻辑关系的组合构成作用域链。作用域链用途是为了保证对执行环境有权访问的所有变量和函数的有序访问。变量标识符的解析是沿着作用域链一级一级地搜索标识符的过程。内部作用域因此可以访问外部作用域。

由于函数调用决定了上下文，在进入一个函数以后，其中声明的变量可以被直接访问（但块作用域中声明的变量要进入块中才可以访问）。也因此，函数作用域就显得相对有点特殊，比起块作用域而言。

2、查看程序运行状态

一个容易想到的问题就是，我当前的程序的局部变量、作用域、上下文和调用堆栈究竟是什么，闭包如何表示？这里就可以借助工具调试直观地观察这个问题的答案了。

整个变量窗格是当前调用堆栈顶部所可以访问到的变量。
• local栏 函数作用域中的变量；
• block栏 块作用域中的变量，可以有多个block栏，即具有父子关系的多个块作用域；
• closure栏 即闭包栏，函数突破自己的作用域限制所能够访问到的外部作用域的变量；
• global栏 全局变量，即顶层对象global上的变量。

其中，值得注意的是：
• 不要被顶部的this迷惑，真实的this即上下文并不一定出现在作用域链最前头（如块作用域、箭头函数）；
• 除了closure栏、global栏和this，其他变量都是声明在当前上下文的；
• closure栏是一个特殊的窗格，函数在调用后进入自己的上下文中，而函数本身又有自己的函数作用域，因此外部作用域所声明的变量都不在当前上下文。当需要访问外部作用域的变量时，需要一个特殊的通道来保证这个操作的正确性，这就是闭包（closure）。

3、作用域链

上文提及到函数作用域有点特殊，这里详细地谈一下为什么。

1）、函数作用域内

对于函数作用域内部而言，因为函数被调用后产生了上下文，所以在ES5及以前内部声明的所有变量的访问都是无拘无束的，因为都在同一层上下文、作用域中（不考虑动态修改作用域的with和catch）。

上下文与函数作用域相对应，作用域链也是函数对象的属性[[scope]]所连接的。ES5及以前的标准的是，作用域链和函数对象上的属性[[scope]]有关。[[scope]]是一个指针数组，始终有一个组员指向了global顶层对象，也因此你可以在任何作用域、上下文内访问到全局变量。它有时候又有一个组员指向closure对象（然而对于作用域链的标准不停地在变，为的就是要既兼容历史又不会有直观上的错误，可能今后又变了），使得其能够通过闭包机制访问自己函数作用域外部的变量。

2）、函数作用域外

由于对于一个特定的函数而言，函数被调用后其内部的函数作用域和上下文属于这个特定的函数。而调用这个函数的函数又有其自身的作用域和上下文，此时我们这里称呼他们为那个特定的函数的“函数作用域外”。

这个特定的函数本身也是一个对象变量，被安置在外层的函数作用域中（先不考虑闭包），即“函数作用域外”是包含这个函数对象变量的。由此，通过这样的递归定义，不难发现所有的作用域都被串联起来了。这就形成了原始的作用域链。

3）、块作用域

原先的设计满足了原先的需求，但是块作用域的出现使得设计需要变更。变更很简单，只需要满足在块作用域声明使用let、const、class和function声明的内容无法在该块作用域外访问就行。也就是说，这个块作用域内部的一些变量依然在当前这个函数被调用产生的上下文，只是限制了一定的规则无法在自身块作用域外被访问了。然后，这并不会改变上面的作用域链，依然可以递归地找到上级、上级。

小结一下，块作用域没有自己的属性[[scope]]，没有自己的上下文（当然也没有this、arguments等对象），也处在它所在的函数被调用后产生的上下文中。它为变量提供了生命周期！

4）、闭包

闭包是一个令人困惑的概念，因为他似乎和离散数学中的理论有点差异。这里有一个定义：闭包是指有权访问另一个函数作用域中的变量的函数。根据定义，闭包是函数，闭包与作用域有关。

闭包难以理解的另一个原因是，其总是和作用域链、上下文、匿名函数、立即执行函数等概念一起出现，使得网络上的示例代码往往充斥着多个语法特性而难以被透彻地解读。显然根据定义，闭包由这些概念中的作用域决定，实际上是函数作用域决定，与上下文有相关性，与匿名函数、立即执行函数和其他概念无关。

由于作用域链以函数作用域为划分基准，函数在被调用后会产生新的上下文，而闭包是突破函数作用域访问外部作用域的函数，需要访问不是自己上下文的变量，因此需要一种机制来保证闭包的正确性。

合理的机制就是，在声明函数的时候，就记录该函数会访问哪些不属于该函数作用域的变量。通过层层递归，不断递交给外层函数那些内层函数闭包变量。即，一个外边的变量的指针可以通过作用域链交给需要这个变量的函数，也就能够表现出内部函数访问外部变量是通过作用域链找到的。

目前的Node.js将闭包变量挂载在[[scope]]上，使得内部函数即便访问诸多层之外的变量也可以一步访问到位。这可以通过调试工具查看得到。在“查看程序运行状态”这节中，就可以通过变量窗格的closure栏查看得到哪些变量是属于闭包传递到当前上下文的。这个closure栏同样可以有多个，代表不同层函数作用域所给予的“便利”。
